# LogosGate（言法界枢）项目思路 v0.2

> The deterministic action-policy firewall for AI agents.
> 面向 Agent 的确定性动作策略防火墙。

## 0. 与 OmniGlyph 的关系

LogosGate 不作为独立项目立即启动，而是先作为 OmniGlyph 的实验性上层模块孵化。

- **OmniGlyph** 负责回答：这个字符、术语、符号是什么？它来自哪里？有没有确定的来源？
- **LogosGate** 负责回答：这个 Agent 准备执行的动作，是否触碰了当前命名空间下的硬性策略边界？

两者的关系是：

```text
OmniGlyph = Symbol / Term Ground Truth Layer
LogosGate = Action Policy Firewall Layer
```

也就是说，OmniGlyph 是“认知事实层”，LogosGate 是“执行边界层”。前者让 Agent 不乱认字、不乱解释术语；后者让 Agent 在真正调用 API、写数据库、发消息、下单、删文件、执行脚本之前，必须通过一层确定性策略校验。

## 1. 核心定位

LogosGate 是一个轻量级、低延迟、可测试的 Agent 动作拦截模块。它不依赖二次大模型判断，不把伦理、安全、业务规则交给概率模型临场发挥，而是把高置信度规则写成可审计的结构化策略数据，在运行时执行确定性校验。

一句话：

```text
LLM 负责生成计划，LogosGate 负责判断计划能不能落地执行。
```

## 2. 为什么需要它？

Agent 系统真正危险的地方，不是模型说错一句话，而是模型说错之后还能继续执行动作。

常见风险包括：

- 自动营销 Agent 为了 KPI 生成刷单、水军、窃取隐私等动作。
- 代码 Agent 在没有确认的情况下删除文件、覆盖配置、泄露密钥。
- 客服 Agent 编造承诺、错误报价、输出未授权合同条款。
- 数据 Agent 对生产数据库执行危险写操作。
- 外贸 Agent 把不确定术语当成确定事实，生成错误报价或合规文件。

Prompt 可以提醒模型，但 Prompt 不是边界。LogosGate 的目标是在动作执行前提供最后一道硬拦截。

## 3. 设计原则

### 3.1 不做“最高真理”，只做“可执行策略”

原始构想中“最高解释权”“绝对真理”等表达具有哲学冲击力，但工程落地时容易过度承诺。LogosGate 的准确定位应是：

```text
source-backed policy-as-data runtime for agent actions
```

中文可以理解为：

```text
面向 Agent 动作的、来源可追溯的策略即数据运行时。
```

它不裁判世界真理，只执行当前组织、场景、命名空间下明确写入的策略规则。

### 3.2 不调用 LLM 做拦截判断

运行时拦截层必须快、稳、可复现。因此 MVP 阶段不做二次 LLM 判断，只使用：

- 精确词匹配
- 正则匹配
- Unicode/术语标准化结果
- 结构化 action 字段
- 命名空间策略表

目标是让拦截逻辑在本地毫秒级完成。

### 3.3 拥抱误差，降低误杀

单纯关键词黑名单会误杀，例如：

```text
“不要刷单” 也包含 “刷单”
```

因此策略规则不能只有 `fatal_keywords`，还需要支持：

- `match_type`: `literal` / `regex` / `term_id` / `action_field`
- `severity`: `block` / `warn` / `review`
- `allow_context`: 允许出现但不应拦截的上下文
- `evidence`: 命中的原文片段和规则来源
- `namespace`: 不同行业、业务线、工具权限下的不同边界

### 3.4 全程可审计

每一次拦截都应返回结构化解释，而不是只抛出一句错误。

拦截结果需要包含：

- 命中的规则 ID
- 命中的策略版本
- 所属命名空间
- 风险等级
- 命中证据
- 推荐处理方式
- 是否允许重试或人工复核

## 4. MVP 范围

第一阶段不要做“世界大词典”，先做一个能跑通闭环的 Agent Action Firewall。

### 4.1 MVP 要做

- 定义 `LogosPolicy` / `LogosRule` 数据模型。
- 支持 JSON/YAML 策略文件加载。
- 提供本地 Python Validator。
- 提供装饰器：`@logos_gate(...)`。
- 提供函数式 API：`validate_action(plan, namespace, concept)`。
- 提供 CLI：`omniglyph logos validate ...`。
- 提供 MCP 工具：`validate_action_policy`。
- 提供 3 个示例策略包：
  - `code_safety`: 文件删除、密钥泄露、危险 shell 命令。
  - `marketing_integrity`: 刷单、水军、虚假数据、隐私窃取。
  - `trade_compliance`: 未授权报价、错误 HS code、合同承诺风险。

### 4.2 MVP 不做

- 不做通用伦理裁判。
- 不做“全人类唯一词典”。
- 不做自动法律判断。
- 不做多轮 LLM 审判。
- 不做自动修改 Agent 输出。
- 不做生产级合规认证。

这些可以作为长期方向，但不能作为第一版承诺。

## 5. 核心数据模型草案

```json
{
  "policy_id": "marketing_integrity.growth.v1",
  "concept": "growth",
  "namespace": "business.marketing",
  "definition": "Growth must increase durable user value or real profit without fabricating demand, violating consent, or creating hidden systemic debt.",
  "version": "0.1.0",
  "rules": [
    {
      "rule_id": "marketing_integrity.no_fake_orders",
      "description": "Block fake orders and paid review manipulation.",
      "severity": "block",
      "match_type": "literal",
      "patterns": ["刷单", "虚假订单", "fake orders", "paid fake reviews"],
      "allow_context": ["不要刷单", "禁止刷单", "avoid fake orders"],
      "source": {
        "type": "project_policy",
        "name": "LogosGate Genesis Policy Pack",
        "version": "0.1.0"
      }
    }
  ]
}
```

## 6. 运行时流程

```text
Human instruction
  ↓
Agent generates action plan
  ↓
OmniGlyph normalizes symbols / terms if needed
  ↓
LogosGate loads namespace policy
  ↓
Validator checks structured action + text evidence
  ↓
Result:
  - allow: execute action
  - warn: continue with logged warning
  - review: require human confirmation
  - block: raise LogosViolationError
```

## 7. API 草案

### 7.1 Python 装饰器

```python
from omniglyph.logos import logos_gate

@logos_gate(namespace="business.marketing", concept="growth")
def execute_marketing_campaign(action_plan: str):
    return submit_campaign(action_plan)
```

### 7.2 函数式 API

```python
from omniglyph.logos import validate_action

result = validate_action(
    text="计划雇佣水军进行刷单，提高转化率。",
    namespace="business.marketing",
    concept="growth",
)

if result.decision == "block":
    raise RuntimeError(result.message)
```

### 7.3 CLI

```bash
omniglyph logos validate \
  --namespace business.marketing \
  --concept growth \
  --text "计划雇佣水军进行刷单，提高转化率。"
```

### 7.4 MCP 工具

工具名：`validate_action_policy`

输入：

```json
{
  "namespace": "business.marketing",
  "concept": "growth",
  "text": "计划雇佣水军进行刷单，提高转化率。"
}
```

输出：

```json
{
  "decision": "block",
  "severity": "block",
  "violations": [
    {
      "rule_id": "marketing_integrity.no_fake_orders",
      "matched": "刷单",
      "evidence": "计划雇佣水军进行刷单，提高转化率。"
    }
  ],
  "message": "Action blocked by business.marketing growth policy."
}
```

## 8. 与 OmniGlyph 当前能力的复用点

LogosGate 不需要重造底层基础设施，可以复用 OmniGlyph 已经完成的能力：

- PyPI 包发布流程。
- GitHub Actions 测试与构建检查。
- MCP stdio server。
- CLI 命令结构。
- SQLite / domain pack 的来源追踪思想。
- `normalize_tokens` 对术语和符号进行执行前标准化。
- `validate_output_terms` 的 guardrail 方向。

## 11. 项目判断

LogosGate 的方向是成立的，但正确落地方式不是一上来做“语义宪法”或“世界唯一法典”，而是先做可测试、可解释、可复用的 Agent Action Firewall。

最小成功标准是：

```text
给定一个 Agent action plan，LogosGate 能在本地毫秒级返回 allow / warn / review / block，并说明命中了哪条规则。
```

只要这个闭环跑通，它就具备真实价值：

- 对开发者：防止代码 Agent 执行危险动作。
- 对业务 Agent：防止营销、报价、合同、客服动作越界。
- 对 OmniGlyph：从符号事实层向执行防护层自然升级。
- 对开源生态：提供一个轻量、可扩展、MCP-ready 的 Agent guardrail 基础组件。

## 12. 最终定位

LogosGate 不是 OmniGlyph 的替代品，而是 OmniGlyph 的上层执行防线。

```text
OmniGlyph sees the symbol.
LogosGate judges the action.
```

中文可以表达为：

```text
万象文枢负责识别符号真值，言法界枢负责守住动作边界。
```
