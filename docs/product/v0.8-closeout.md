## What This Batch Adds

### Policy Pack intent guardrails

- Adds the local Policy Pack format with `policy.json` and `intents.csv`.
- Adds Policy Pack init, validation, loading, manifest conversion, and API/MCP path restriction helpers.
- Keeps inline intent manifests supported for compatibility.
- Adds CLI commands:
  - `omniglyph init-policy-pack`
  - `omniglyph validate-policy-pack`
  - `omniglyph enforce-intent`
- Adds API endpoint:
  - `POST /api/v1/policy/validate-pack`
- Adds MCP tool:
  - `validate_policy_pack`
- Extends intent enforcement through `policy_pack_path` for API and MCP.

### Intent parameter validation

- Adds dependency-free runtime validation for Policy Pack and inline manifest `parameters_schema`.
- Supported deterministic subset:
  - `type`
  - `required`
  - `properties`
  - `enum`
  - string length bounds
  - numeric bounds
  - array `items`
- Invalid parameters return `decision: "block"` and `status: "invalid_parameters"` with `parameter_findings`.
- Unsupported schema keywords are ignored rather than executed.

### Post source-batch fail-closed hardening (2026-07-16)

Subsequent hardening on `codex/parameter-schema-fail-closed` (not part of the original source-batch merge set; see dedicated closeouts):

- Intent-policy fail-closed: invalid Policy Packs cannot load; invalid inline manifests return `invalid_manifest` evidence.
- Parameter-schema fail-closed: documented keyword values are meta-validated; deep/cyclic schemas and values are evaluated iteratively.
- Full closeout reports:
  - `docs/superpowers/reviews/2026-07-16-intent-policy-fail-closed-closeout.md`
  - `docs/superpowers/reviews/2026-07-16-parameter-schema-fail-closed-closeout.md`
- Latest observed release gate on that branch: `263 passed`.

### Output guardrail policy modes

- Adds optional policy modes for unknown, unapproved, and secret output terms.
- Supported actions:
  - `allow`
  - `review`
  - `block`
- Strict blocking remains the default behavior when no policy override is provided.
- API, MCP, and Python callers share the same runtime policy path.

### Guardrail Review Packet

- Adds deterministic `review_packet` evidence when risky output terms are present.
- Groups risky terms by:
  - `unknown`
  - `unapproved`
  - `secret`
- Summaries include term count, group count, actions, and classes.
- Packet hardening adds branch coverage for unapproved terms and policy-allowed unknown terms.
- `review_packet` is omitted for fully safe output.

### Output guardrail CLI

- Adds `omniglyph enforce-output` as a local JSON evidence command.
- Supports repeated `--term`, optional `--policy`, and optional `--actor-id`.
- Latest online readiness hardening also exposes local CLI wrappers for existing evidence surfaces:
  - `omniglyph validate-output`
  - `omniglyph scan-language-input`
  - `omniglyph scan-output-dlp`
  - `omniglyph audit-explain`
- Example:

```bash
omniglyph enforce-output --term FOB --term "HS 7604.99X" --policy '{"unknown_action":"review"}'
```

## Operational Notes

- This was originally closed as a source-ready feature batch.
- Release-prep now uses package version `0.8.0b0`.
- Before package publication, run the full release gate from the current source candidate and upload to TestPyPI first.
- Run `scripts/release_check.sh` before PR merge, tag creation, TestPyPI/PyPI upload, or MCP registry update.
- Regenerate `dist/` immediately before any upload; do not upload stale local build artifacts.
- Upload exact artifact filenames rather than `dist/*`.
- `OMNIGLYPH_POLICY_PACK_ROOT` restricts API/MCP Policy Pack paths. The CLI remains local-operator oriented.
- Policy Packs and output guardrails return evidence only. They do not execute commands, rewrite output, persist review queues, or call external approval systems.

## Explicit Boundaries

This batch does not add:

- automatic output rewriting
- persistent review queues
- database ingestion for Policy Packs
- ERP, email, webhook, or ticketing integrations
- automatic shell or tool command execution
- complete prompt-injection immunity
- package registry publication
- MCP registry publication

Host systems remain responsible for reading the JSON evidence and deciding whether to stop, route, approve, regenerate, or continue.
